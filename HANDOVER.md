# 画像・プロンプト共有ボード 引き継ぎドキュメント

Stable Diffusion 等で生成した画像と、そのプロンプト情報を仲間内で共有するための WEB アプリ。本ドキュメントは現状の仕様・設計思想・運用方法をまとめた引き継ぎ資料である。

---

## 1. 全体構成

| 要素 | 実体 |
|---|---|
| フロントエンド | `index.html` 1ファイル(CSS / HTML / JS すべて内包)。静的ホスティング(GitHub Pages 等)に置く想定 |
| バックエンド | `Code.gs`(Google Apps Script)。ウェブアプリとしてデプロイ(実行: 自分 / アクセス: 全員) |
| データベース | Google スプレッドシートの `posts` シート 1枚 |
| 画像ストレージ | Google Drive の指定フォルダ(`Code.gs` の `FOLDER_ID`) |

`index.html` 冒頭付近の `GAS_URL` に GAS ウェブアプリの URL を設定して接続する。

### 通信方式(重要)
- **読み取り(全投稿取得)**: GAS ウェブアプリへのクロスオリジン fetch は CORS でブロックされることがあるため、**JSONP**(`<script>` タグ読み込み、`?callback=` パラメータ)で取得する。`doGet` は callback があれば JavaScript として、なければ素の JSON を返す。
- **書き込み(投稿・いいね・DL カウント)**: `fetch` の POST。`Content-Type: text/plain;charset=utf-8` で JSON 文字列を送る(プリフライト回避のため)。`doPost` が `body.action` で分岐: `like` / `download` / それ以外は新規投稿。
- 書き込みのカウント更新は `LockService` で排他制御(`bumpCount`)。

## 2. データ設計

`posts` シート 1枚に**画像投稿とプロンプト例投稿を混在**させる。列構成(`COLUMNS`、この順):

```
id, timestamp, imageUrl,
title, author, prompt, negativePrompt,
settingsJson, tags, memo,
shareId, downloadCount, likeCount,
type, basePromptId, parentId
```

- `id`: UUID。`shareId`: 共有 URL 用の 6 文字コード。
- `settingsJson`: 生成設定のオブジェクトを JSON 文字列で保存。
- `tags`: カンマ区切り文字列。
- `type`: `'image'` または `'prompt'`。**空欄は `'image'` 扱い**(列追加前の既存データとの後方互換)。
- `basePromptId`: 画像投稿が「ベースにしたプロンプト例」の id。
- `parentId`: 画像投稿の親作品(バリエーション元)の id。

フロントは全件を `posts` に読み込み、`imagePosts` / `promptPosts` に派生させて使う。

### 投稿の 2 種類と 2 つの紐づけ
1. **画像投稿**(type=image): 画像 + タイトル / 投稿者 / プロンプト / ネガティブ / メモ / 生成設定 / タグ。
2. **プロンプト例投稿**(type=prompt): 画像なし。自作プロンプトを作品として共有する(タイトル / 投稿者 / プロンプト / ネガティブ / メモ / 生成環境 / タグ)。ダウンロード機能はなく、いいねのみ。

紐づけは 2 系統あり、目的が異なる:
- **ベースプロンプト紐づけ**(画像 → プロンプト例): 「このプロンプト例を参考に作った」という関係。プロンプト例の詳細下部に、それをベースにした画像作品が一覧表示される。プロンプト例を**タグのように使って作品を探す**ための仕組み。
- **親子紐づけ**(画像 → 画像): 同じ設定で Seed 違いを量産したとき用。子は**画像投稿一覧に表示されず**、親の詳細下部に「この作品のバリエーション」として一覧表示される。**一覧が同種の画像で溢れるのを防ぐ**ための仕組み。親の候補は「親を持たない画像投稿」のみ(孫は作れない = 階層は 1 段まで)。プロンプト例詳細の派生一覧・派生件数からも子は除外する(溢れ防止の一貫性)。

## 3. 画面構成(タブ)

上部タブで 4 画面を切り替える(`switchTab`)。

### 3-1. 投稿タブ
- 上部のモード切替ボタンで「画像を投稿」(デフォルト)と「プロンプト例を投稿」を切替(`currentPostMode`)。
- **画像フォーム**: ドロップゾーンに画像を置くと、PNG の `parameters` チャンクや JPEG/WebP の EXIF `UserComment` から生成情報を解析し、プロンプト・ネガティブ・生成設定を**自動入力**する(`PARAM_KEY_MAP` で AUTOMATIC1111 系のキーをフォーム項目にマッピング)。手動修正可。
- 生成設定は `SETTINGS_FIELDS` 定義(UI Preset / Checkpoint / Model hash / VAE / Sampling Method / Steps / Width / Height / CFG Scale / Seed / Version 等 19 項目)からグリッド生成。既存投稿の値を datalist で補完候補に出す。
- **ベースにしたプロンプト例(任意)**: モーダルピッカー(検索 + ソートチップ)で選択。選択すると**そのプロンプト例のタグが選択タグに自動追加**される(重複なし。解除してもタグは消さない)。
- **親にする投稿作品(任意)**: 同様のモーダルピッカー(サムネイル付き、検索 + ソートチップ)。候補は親を持たない画像投稿のみ。
- タグ: 既存タグのクラウド(クリックで選択 / 2 行超は「全て表示」でモーダル展開)+ 新規タグ入力。**タグは画像投稿とプロンプト例で共通のプール**(全投稿から集約)。
- **プロンプトフォーム**: 画像関連を除き画像フォームと同等(接頭辞 `p` の要素群、例 `pTitleInput`)。

### 3-2. 画像投稿一覧タブ
- 表示形式: リスト / パネル(メイソンリー風グリッド、`layoutPanelGrid` で縦横比を反映)。パネル表示で行クリックするとまずフルスクリーン画像モーダル。
- ソートチップ: 日時 / いいね / ダウンロード / 投稿者 / UI Preset / Checkpoint。クリックで 降順▼ → 昇順▲ → 解除 を循環し、複数選択でクリック順が優先順位(バッジ表示)。
- 絞り込み: 投稿者 / UI Preset / Checkpoint のテキスト部分一致 + タグクラウド(AND / OR / NOT 切替)。
- **親を持つ投稿(子)は表示されない。**

### 3-3. プロンプト投稿一覧タブ
- 画像一覧と同等の機能をプロンプト例向けに並列実装(リスト表示のみ)。ソート: 日時 / いいね / 投稿者 / Checkpoint。絞り込み: 投稿者 / Checkpoint + タグ(AND / OR / NOT)。
- 各行にプロンプト冒頭の抜粋と「派生作品 N 件」を表示。

### 3-4. 本ツールの使い方タブ
概要と Q&A(削除は管理人のみ・削除依頼の出し方 / 個人情報の扱い / 画像投稿とプロンプト例の違い / 派生作品の探し方 / 親子紐づけの使い方 / メタデータ自動入力 / 編集不可 / いいねの仕様)。

### 詳細モーダル(共通の `modalOverlay` / `modalBody` を使い回す)
- **画像詳細**: 画像(クリックで原寸系表示)、いいね、ダウンロード(原寸取得 + DL カウント)、共有、PROMPT / NEGATIVE コピー、メモ、設定表、「ベースにしたプロンプト例」リンク、「親作品」リンク、下部に子バリエーション一覧。
- **プロンプト例詳細**: いいね、共有、PROMPT / NEGATIVE コピー、メモ、生成環境、下部に「このプロンプト例をベースにした投稿作品」一覧(子は除外)。
- モーダル内の一覧行はクリックでその投稿の詳細に遷移する(`wireImageRows(container, { detailOnly: true })`)。

## 4. その他の仕様

- **いいね**: 1 投稿につき 1 端末(ブラウザ)1 回。`localStorage` の `likedPosts` で管理し、再クリックで取り消し。楽観更新 → サーバー結果で補正。
- **共有 URL**: `?p=<shareId または id>`。読み込み後 `openSharedFromUrl` が該当投稿の詳細を直接開く(type によって画像 / プロンプトのタブと詳細を出し分け)。
- **画像 URL**: Drive の `thumbnail?id=...&sz=w◯◯◯` 形式でサイズ別に取得(`imageSrcFor`)。一覧 w320 / パネル w640 / 詳細 w1000、DL は `lh3.googleusercontent.com/d/<id>=s0`(原寸)。
- **削除・編集機能はない**。削除はスプレッドシートの行を管理人が手で消す運用(Q&A に明記)。
- モーダル類は ESC キー / 背景クリックで閉じる。
- 日本語ソートは `localeCompare(..., 'ja', { numeric: true })` ベース(`sortJa` / `compareWithKeys`)。

## 5. 設計思想

1. **利用者が操作で迷わないことを最優先**。既存のデザイン・レイアウト・操作感を壊さない。新機能も既存 UI の文法(モーダル、チップ、タグクラウド、行スタイル)を流用して増やす。
2. **一覧を溢れさせない**。似た作品の量産は親子紐づけで畳み、プロンプト派生一覧からも子を除外する。
3. **後方互換**。シート列は末尾追記のみ。新列が空でも既存データが従来どおり動く既定値を必ず持たせる(`type` 空 = image 等)。
4. **単一ファイル完結**。フロントは `index.html` 1 ファイルに CSS / HTML / JS を収める。外部依存なし。
5. **既存コード非破壊・最小変更**。機能追加は既存関数を書き換えるより、共通ロジックを関数抽出(例: `compareWithKeys`, `imageRowHtml`, `wireImageRows`, `renderPickerSortChips`)して並列実装する。
6. **コメント方針**: 説明コメントは書かない。セクション見出しコメント(`// ===== X =====` や `// PARENT POST PICKER` 形式)のみ許可。
7. **命名規則**: プロンプト側の並列実装は接頭辞 `p`(`pSortKeys`, `pTagCloud` 等)。ピッカー系は用途名(`basePrompt◯◯`, `parent◯◯`)。
8. **タグは全投稿共通のプール**。画像とプロンプト例で分けず、`allUniqueTags()`(全 `posts` 起点)に集約する。

## 6. デプロイ・運用手順

### 初回セットアップ
1. Google スプレッドシート新規作成 → 拡張機能 → Apps Script に `Code.gs` を貼り付け。
2. `FOLDER_ID` に画像保存先 Drive フォルダの ID を設定。
3. 関数プルダウンで `setup` を実行(権限承認 → `posts` シートとヘッダー生成)。
4. デプロイ → 新しいデプロイ → ウェブアプリ(実行: 自分 / アクセス: 全員)→ URL を `index.html` の `GAS_URL` に設定。
5. `index.html` を静的ホスティングに配置。

### 更新時(毎回必須)
- `Code.gs` を変更したら「デプロイを管理」→ 鉛筆 → バージョン「**新バージョン**」→ デプロイ。**新しいデプロイを作ると URL が変わるので既存デプロイの更新を使う。**
- シート列を追加したら `setup` を再実行(ヘッダー行を `COLUMNS` で上書きするだけで、データ行は触らない)。
- `index.html` はファイル差し替えのみ。

### 変更時の検証(最低限)
- `index.html` の `<script>` を抽出して `node --check`(構文)。
- JS が `getElementById` / `querySelector('#...')` で参照する id が HTML(動的生成含む)に存在するか突合。

## 7. 既知の制約・注意点

- 投稿の編集・投稿者自身による削除は不可(管理人がシートを直接操作)。
- いいねは端末 / ブラウザ単位のため、別端末からは別カウント。
- 親子は 1 階層のみ(子を親には選べない)。既存投稿を後から子に変える機能はない。
- 紐づけ(basePromptId / parentId)は投稿時のみ設定でき、後から変更できない。
- GAS ウェブアプリの特性上、シートが大きくなると `doGet` の全件取得が遅くなる(現状ページングなし)。
- 画像は Drive の共有設定「リンクを知っている全員が閲覧可」に自動設定される(失敗しても投稿自体は成立)。
